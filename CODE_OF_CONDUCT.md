# MMMail Code of Conduct

## Scope

This code of conduct applies to MMMail project spaces: issues, pull requests, discussions, documentation, community channels, and maintainer-managed release coordination.

MMMail follows Contributor Covenant 2.1 with project-specific clarifications for security disclosure, self-hosted operations, and release governance.

## Expected Behavior

- Be direct, respectful, and technical.
- Assume good intent, but require evidence for technical claims.
- Keep disagreement focused on code, architecture, documentation, security impact, or user impact.
- Respect privacy. Do not publish private vulnerability details, credentials, private contact details, or non-public customer information.
- Help keep the project usable for self-hosted operators and external contributors.

## Unacceptable Behavior

- Harassment, threats, personal attacks, or discriminatory language.
- Publishing secrets, exploit details, private reports, or production data.
- Repeatedly derailing technical threads after a maintainer decision is recorded.
- Misrepresenting MMMail support, commercial promises, licensing, or release status.

## Enforcement

The BDFL and release owner may hide comments, close threads, request edits, block participants, or reject contributions that violate this policy. Security-sensitive violations may be handled privately first.

Report conduct concerns through the private path documented in `SECURITY.md`. If private vulnerability reporting is not enabled, use the minimal security contact request issue template without including sensitive details.

## Enforcement Guidelines

Maintainers should choose the smallest action that protects the project and affected participants:

1. Correction for minor, first-time issues.
2. Warning for repeated or disruptive behavior.
3. Temporary restriction for sustained violations or ignored warnings.
4. Permanent restriction for harassment, threats, privacy breaches, or intentional publication of secrets or exploit details.
