# Mail Zero-Knowledge Roadmap

**版本**: `mail-zero-knowledge-roadmap-v1`
**日期**: `2026-04-02`
**作者**: `Codex`

## 目标
- 为 `Mail E2EE foundation + message encryption` 之后的演进提供真实、可执行、可审计的零知识路线。
- 明确当前仓库已经做到什么、还没做到什么，以及每一阶段的前置依赖。
- 避免把 `Mail E2EE` 当前能力误写成“服务端无法读取任何用户邮件”的零知识模型。

## 当前已交付基线
### 已存在能力
1. 客户端可生成并保存 `Mail E2EE key profile`。
2. 写信页可基于真实 recipient readiness，在 `READY` 的内部路由场景下本地加密正文。
3. 邮件详情页可要求用户输入 passphrase，在浏览器内本地解密正文。

### 当前仍不是零知识的原因
1. 服务端仍可处理非 E2EE 邮件明文。
2. 草稿、附件、搜索索引、预览摘要与路由元数据仍未进入零知识模型。
3. 私钥保护仍依赖单独 passphrase，而不是用户主认证秘密驱动的完整密钥层级。
4. 设备同步、恢复、吊销、轮换、审计尚未建立。

## 非目标
- 本文档不宣称 `v1.2` 已交付零知识邮件服务。
- 不把 `Mail E2EE` 当前正文加密闭环扩展成附件、草稿、Drive 或 Calendar 的已上线承诺。
- 不直接定义 `SMTP / IMAP / Bridge` 的实现计划；协议栈另见独立 discovery 文档。

## 设计原则
- **客户端持钥**：私钥生成、解锁、内容加解密都必须在客户端完成。
- **服务端不持明文解密能力**：服务端只保存公开资料或密文包，不持有可直接恢复明文的秘密。
- **分阶段替换**：先把最小主路径做真，再逐步迁移草稿、附件、检索与多设备能力。
- **显式边界**：每一阶段都要说明仍暴露给服务端的内容，禁止营销式表述。

## 分阶段路线
### Phase 0 - 当前基线（已完成）
- `Mail E2EE foundation`
- `recipient discovery`
- `READY` 内部路由的正文加密发送
- 详情页本地解密

### Phase 1 - Key hardening
**目标**：把“加密私钥包上传”升级为更清晰的客户端密钥层级。

建议项：
1. 引入基于 `Argon2id` 的 `mail secret` 导出策略，用于本地保护私钥包。
2. 将当前独立 passphrase UI 与账户主认证关系明确拆开：
   - 允许独立 mail secret
   - 或引入设备侧派生与二次确认流程
3. 增加密钥轮换元数据：
   - `keyVersion`
   - `supersededAt`
   - `revokedAt`
4. 为后续多设备同步预留 `device-wrapped session` 结构，但本阶段不开放自动同步。

### Phase 2 - Content envelope expansion
**目标**：把内容加密从“正文单字段”扩展到完整邮件实体。

建议项：
1. 草稿内容改为客户端 envelope 加密。
2. 附件使用独立 content key，邮件元数据只保存加密后的 key reference。
3. 预览摘要不再由服务端直接读取正文生成，而改为：
   - 客户端本地缓存摘要
   - 或服务端仅保存显式授权的弱摘要字段
4. 发送路径必须同时生成 sender-readable 副本，避免发件人无法回看已发送内容。

### Phase 3 - Zero-knowledge mailbox operations
**目标**：把收件箱操作逐步迁移到不依赖服务端明文的模型。

建议项：
1. 搜索从明文索引切换到受限能力模型：
   - 客户端索引
   - 或受控关键词 tokenization
2. 标签、星标、已读等状态继续可由服务端保存，但正文、附件、草稿保持密文。
3. 引入密钥吊销、恢复包、审计记录与设备授权列表。
4. 明确哪些元数据仍由服务端可见：
   - 发件时间
   - 路由信息
   - mailbox ownership
   - anti-abuse 所需最小审计字段

### Phase 4 - Cross-product expansion
**目标**：在 Mail 路径稳定后再评估 `Drive / Calendar / Docs`。

前置条件：
1. Mail 的密钥生命周期和恢复机制已稳定。
2. 客户端大文件加密上传链路已验证。
3. 搜索、协作与预览体验有明确降级方案，而不是静默失败。

## 关键架构议题
### 1. 用户认证秘密与邮件加密秘密的关系
- 若直接复用登录密码，需处理服务端认证与客户端派生边界。
- 若独立维护 mail secret，需解决 UX、恢复与设备 onboarding 复杂度。

### 2. 发送者自读副本
- 当前正文加密已经把 sender 自己加入加密收件方。
- 后续附件、草稿、线程回复也必须沿用同一原则，否则发件箱不可用。

### 3. 搜索与预览
- 零知识与服务端全文检索天然冲突。
- 必须提前定义哪些能力接受变弱，哪些能力需要新的客户端索引方案。

### 4. 多设备与恢复
- 没有恢复机制的零知识系统在真实部署中不可接受。
- 恢复包、设备授权与吊销机制必须在进入正式承诺前完成设计与验证。

## 风险
### 安全风险
- 若在服务端加入任何“临时解密”后门，会直接破坏零知识前提。
- 若客户端把 passphrase / mail secret 落盘到本地存储或日志，会破坏密钥边界。

### 产品风险
- 搜索、预览、草稿自动保存、Web 端多标签页恢复都会受到影响。
- 若不先完成 UX 说明，用户会把“受限能力”误判为 bug。

### 工程风险
- 数据迁移涉及历史明文邮件、草稿与附件；不能在单个版本中一刀切切换。
- 需要跨前端、后端、迁移脚本、运维备份与 threat model 联动推进。

## `v1.2` 结论
- `v1.2` 当前已具备 `Mail E2EE` 的真实基础路径，但**不是零知识邮件架构**。
- `Zero-knowledge roadmap` 在 `v1.2` 的交付物是路线、阶段拆分与边界文档，而不是功能承诺。
- 进入后续版本前，必须先完成威胁模型更新、迁移策略、恢复机制与 UX 评审。
