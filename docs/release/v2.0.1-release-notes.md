# MMMail v2.0.1 Release Notes

Prepared as a focused hardening follow-up to the shipped `v2.0.0` baseline.

## Highlights
- scope-aware settings panels and system health integration in `frontend-v2`
- stabilized public-share, workspace aggregation, platform, AI, and MCP capability contracts
- explicit `frontend-v2` and backend contract suites added to local and CI release hardening gates

## Validation
- `pnpm --dir frontend-v2 test`
- `mvn -f backend/pom.xml -pl mmmail-server -am -Dtest=PlatformCapabilityIntegrationTest,PublicShareCapabilityIntegrationTest,WorkspaceAggregationIntegrationTest,AiMcpCapabilityIntegrationTest,RequestHeaderContractIntegrationTest,ObservabilityIntegrationTest -Dsurefire.failIfNoSpecifiedTests=false test`
- `bash scripts/validate-local.sh`

## Included
- `frontend-v2` settings panels now remain aligned with active scope and expose system health / integrations verification surfaces
- backend capability contracts for public-share, workspace aggregation, platform, AI, and MCP remain pinned to the v2.0.1 payload shape
- release hardening gates now explicitly execute the shipped `frontend-v2` contract regression suites and targeted backend contract regressions

## Upgrade Notes
- `v2.0.1` stays within the `v2.0.0` shipped boundary and does not add new top-level route or deployment model claims
- `/api/v1/system/health` remains the supported system health endpoint for the admin verification flow in this release
- existing `v2.0.0` release notes remain the source of truth for original shipped boundary claims

## Ops Handoff / Runbook Pointers
- `docs/deployment-runbook.md`
- `docs/ops/runbook.md`
- `docs/ops/install.md`
- `docs/ops/upgrade.md`
- `docs/ops/backup-restore.md`

## Known Limitations
- `v2.0.1` is a hardening release and does not promote preview labs surfaces into new GA top-level products
- this release does not expand the deployment topology, tenant isolation model, or messaging architecture beyond the shipped `v2.0.0` baseline
