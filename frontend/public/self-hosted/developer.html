Developer Station / 开发者文档站

This browser station is the self-hosted handoff layer for implementation owners. It aligns the shipped mainline, team rollout, identity readiness, and current backend contract before anyone promises unsupported capability or wires against the wrong deployment target.

What this station is / 这是什么

  • A browser hub for current Community docs and backend contract entry points.
  • A deployment-scoped handoff for operators and developers working on the same self-hosted environment.
  • Not a hosted developer portal, SDK guarantee, or enterprise feature promise.

Priority order / 当前优先级

  • P0 — keep implementation work anchored to the shipped Mail → Calendar → Drive → Pass mainline.
  • P1 — treat Drive E2EE delivery depth and Pass handoff clarity as the next mainline quality layer, not as separate Labs work.
  • P2 — keep identity readiness and developer docs explicit so enterprise asks do not get misrepresented as shipped capability.
  • P3 — do not route current rollout effort into non-mainline Preview engines.

Suggested workflow / 推荐流程

  1. Open the adoption guide and confirm the current mainline and support boundary.
  2. Review team enablement so the first rollout task matches a real Mail → Calendar → Drive → Pass handoff.
  3. Review identity readiness before anyone implies enterprise sign-in or provisioning exists today.
  4. Point this station at the real backend origin, then branch into Swagger UI, OpenAPI JSON, and the API guide.
  5. Keep unsupported areas explicit: no hosted portal promise, no SDK guarantee, no enterprise identity automation claim.

Current boundary / 当前边界

  • Community v1.6.1 ships a browser developer station, not a public developer platform.
  • `SMTP inbound / IMAP / Bridge` and enterprise identity automation are still outside the shipped mainline.
  • `Pass` is visible in default navigation, but remains `Beta` and should be described that way in developer-facing docs.
