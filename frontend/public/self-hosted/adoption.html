MMMail Community v1.6.1

Adoption Guide / 采用指引

This quick page explains the current self-hosted mainline for Community: Mail → Calendar → Drive → Pass. It is not a hosted onboarding portal, and it does not claim SMTP inbound / IMAP / Bridge, zero-knowledge mail, or enterprise identity automation.

Mainline definition / 主线定义

  • Mail — establish the protected context, recipients, and subject.
  • Calendar — turn that context into an internal checkpoint with RSVP visibility.
  • Drive — deliver the encrypted file, artifact, or recovery package.
  • Pass — hand over the final credential, secure link, alias route, or secret detail.

Current priority order / 当前优先级

  • P0 — deepen the shipped team mainline until Mail, Calendar, Drive, and Pass form one continuous handoff.
  • P1 — keep pushing Drive E2EE delivery depth and keep Pass visible as the default-nav Beta handoff surface.
  • P2 — keep identity readiness, developer docs, and post-deploy enablement explicit without turning them into shipped enterprise promises.
  • P3 — keep non-mainline Preview modules outside the default rollout promise while pluginization / externalization stays future work.

Launch tracks / 启动轨道

Administrator baseline

  • Generate the Mail E2EE key profile and recovery package.
  • Confirm the release boundary before promising capabilities to the team.

Team first collaboration

  • Run one real handoff through Mail, Calendar, Drive, and Pass in order.
  • Use shipped web surfaces only; keep Beta boundaries explicit.

Identity readiness

  • Keep SSO / SCIM / LDAP as readiness topics, not shipped login promises.
  • Prepare IdP ownership, fallback admins, and rollout expectations before enterprise requests arrive.

Developer handoff

  • Align the operator sequence before wiring integrations.
  • Use Swagger UI and OpenAPI from the actual deployment target.

Recommended proof sequence / 推荐验证顺序

  1. Open Settings and complete the Mail E2EE and PWA baseline.
  2. Open Suite Overview and run the mainline once end to end.
  3. Use the team guide to capture pilot roles, proof points, and rollout sequence.
  4. Use Drive or Pass to prove one secure handoff that another teammate can consume.
  5. Review the identity readiness page so SSO / SCIM / LDAP requests stay aligned with current scope.
  6. Give developers the API page and OpenAPI schema for the running backend.

Current boundary / 当前边界

Community v1.6.1 focuses on mainline definition, collaboration depth, and adoption experience. It still does not ship enterprise identity automation, browser extension autofill, or non-mainline preview engines as part of the default support promise.
